# 调试工作流

高效调试不是“让 AI 猜问题”，而是把证据喂给它。

## 标准四段式

1. 复现：拿到稳定复现步骤
2. 定位：缩小故障范围
3. 修复：最小改动修复
4. 回归：验证不引入新问题

## 第 1 步：先给可复现信息

至少提供：

- 报错原文
- 触发命令
- 环境信息（Node 版本、依赖版本）
- 最近相关改动

示例提示：

```text
这是可复现的错误日志，请先分析最可能的 3 个根因，
并给出最小排查顺序，暂时不要改代码。
```

## 第 2 步：按假设逐个验证

让 OpenCode 每次只验证一个假设，避免多变量同时改动。

## 第 3 步：要求最小修复

```text
请给最小修复方案，只改必要文件；
修复后运行相关测试并解释为什么这个改动能解决问题。
```

## 第 4 步：回归验证

1. 原始问题不再出现
2. 相邻功能不被破坏
3. 核心路径测试通过

## 常见反模式

1. 直接贴一句“项目跑不起来”
2. 同时改十几个文件再验证
3. 不跑回归就提交

## 调试模板

```text
问题：...
复现步骤：...
期望行为：...
实际行为：...
错误日志：...
先做：根因排序 + 排查计划（不改代码）
再做：最小修复 + 验证 + 风险说明
```
