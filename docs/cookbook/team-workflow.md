# 团队协作工作流

团队用 OpenCode 时，关键不是“谁更会写提示词”，而是“流程是否可复制”。

## 推荐角色分工

1. 需求负责人：定义范围与验收标准
2. 执行负责人：驱动 OpenCode 计划与改动
3. 审查负责人：看 diff、跑回归、做风险确认

三角色可以是三个人，也可以两人兼任，但职责要清晰。

## 标准协作节奏

1. 需求评审：确定范围、时间、风险
2. Plan 评审：只看计划，不看实现
3. Build 执行：按计划分批落地
4. PR 审查：聚焦风险和回归
5. 复盘沉淀：模板与规则更新

## 团队规则建议

1. 每次任务先 plan 后 build
2. 单次 PR 保持可审查规模
3. 必须附验证命令结果
4. 高风险目录双人审查

## 推荐 PR 模板字段

- 目标与范围
- 改动文件
- 验证命令与结果
- 已知风险
- 回滚方式

## 常见组织问题

### 问题 1：不同人风格差异太大

做法：统一规则模板、PR 模板、验证门禁。

### 问题 2：速度快但 bug 多

做法：强制分步执行与每步验证。

### 问题 3：新人上手慢

做法：提供 3 个固定提示模板：

1. 计划模板
2. 执行模板
3. 修复模板

## 团队落地最小包

1. 一份项目规则文件
2. 一份 OpenCode 提示词模板集
3. 一份 PR 模板
4. 一条统一验证命令（如 `pnpm verify`）

先把这四件事做好，团队效率会明显提升。
