# 基本工作流

一个高效、可控的 Opencode 工作流，通常包含以下 5 个阶段。

## 1. 明确任务边界

写清楚：

- 要改什么
- 不要改什么
- 验收标准是什么

提示词越具体，结果越稳定。

## 2. 先计划再执行

先让 Opencode 输出计划，再确认执行顺序。典型提示：

```text
先不要改代码。请先给出执行计划和潜在风险。
```

## 3. 小步快跑

把任务拆成多个小步骤，每步结束即验证，避免一次改太多难以回退。

## 4. 命令验证

至少执行：

- 测试
- 构建
- 关键路径手测

不要只看“它说成功了”，要看真实命令输出。

## 5. 总结与沉淀

让 Opencode 输出：

- 变更摘要
- 风险点
- 后续建议

然后将可复用提示词沉淀到团队文档中。

## 常见反模式

- 任务描述过于模糊
- 不限制改动范围
- 不跑测试直接提交
- 把 AI 输出当成最终结论

## 推荐模板

```text
目标：...
范围：只允许修改 ...
约束：不得修改 ...
验收：必须通过 ...
输出：先给计划，再逐步执行，每步汇报改动与验证结果
```

将以上模板长期复用，你会明显降低返工率。
